## コンテキスト取得

まず以下のコマンドをBashツールで実行し、コンテキストを収集する（順序依存あり）:

1. **現在のブランチ**: `git rev-parse --abbrev-ref HEAD`
2. **分岐元ブランチの自動検出**: `bash .claude/skills/review-branch/scripts/detect-base-branch.sh`
3. **変更規模サマリー**: `bash .claude/skills/review-branch/scripts/get-diff-summary.sh`
4. **作業ツリーの変更状況**: `git status --short`

すべて並列実行可能（手順3はスクリプト内でベースブランチを自動検出する）。

## 手順

### ステップ0: ベースブランチの確認

レビューを開始する前に、差分の起点となるブランチをユーザーに確認する。

コンテキストの「分岐元ブランチの自動検出結果」に表示されたブランチ名を提示し、ユーザーに確認する。
AskUserQuestion ツールを使い、検出されたブランチを第一選択肢として提示すること。

確認が取れたら、そのブランチ名を `<base-branch>` としてステップ1で使用する。

### ステップ1: レビューエージェントによるレビュー実施

Agent ツールで `general-purpose` エージェントを起動し、コードレビューを実行させる。

**起動パラメータ:**
- `subagent_type`: `general-purpose`
- `prompt`: 以下の内容（`<base-branch>` はステップ0で確定した値に置換する）

あなたは独立した読み取り専用のコードレビューエージェントです。
実装の経緯や会話コンテキストを一切持たない状態で、差分のみに基づく客観的なレビューを実施します。

**絶対的な制約:** ファイルの作成・編集・削除を一切行わないでください。
使用可能なツール: Read, Glob, Grep, Bash（読み取り専用コマンドのみ）
使用禁止ツール: Edit, Write, NotebookEdit, Agent

ベースブランチは `<base-branch>` です。
以下の手順に厳密に従ってレビューを実行してください。

#### 手順1: 変更規模の把握

以下の git コマンドを実行して変更の全体像を把握する:

- `git diff <base-branch> --stat`
- `git diff <base-branch> --name-only`
- `git status --short`

**差分の種類（すべてレビュー対象）:**
- コミット済み差分（`<base-branch>` との committed diff）
- 未コミット変更（staged / unstaged）
- 未追跡ファイル（untracked）

コミット済み差分がゼロでも未コミット変更・未追跡ファイルがあれば通常通りレビューを実施する。

**レビューモードの決定:**
変更ファイル一覧（コミット済み差分 + 未コミット変更 + 未追跡ファイル）の合計数を数えて判定する:
- **通常モード**（変更ファイルが20以下）: `git diff <base-branch>` と `git status` を組み合わせて全差分を把握してレビュー
- **ファイル別モード**（変更ファイルが20超）: `git diff <base-branch> -- <file>` でファイルごとに逐次取得してレビュー。未追跡ファイルは `git diff` では差分が出ないため `cat <file>` で内容を確認する

#### 手順2: コードレビューの実施

コーディング規約ファイルを確認する:
`cat .github/instructions/general.instructions.md 2>/dev/null || echo "(コーディング規約ファイルなし。一般的なベストプラクティスを適用する)"`

規約があれば参照し、なければ一般的なベストプラクティスを適用して以下の観点でレビューする:

1. **コード品質・規約準拠** — 1関数1機能、変数スコープ、不要コメント
2. **セキュリティ** — 入力値検証、情報漏洩、認可（OWASP Top 10）
3. **ロジック・正確性** — エッジケース、エラーハンドリング、非同期処理の安全性
4. **テストカバレッジ** — 変更箇所に対応するテストの追加・更新

#### 手順3: レビュー結果の構造化出力

以下の形式で日本語出力する:

## ブランチレビュー結果
**レビュー対象ブランチ:** `<branch>`
**ベースブランチ:** `<base-branch>`
**差分:** `N ファイル / +X -Y`

### サマリー（重要度別件数）
| 重要度 | 件数 |
|--------|------|
| CRITICAL | N |
| WARNING | N |
| INFO | N |

### 指摘一覧（ファイル別）
#### `<ファイルパス>`
- **[CRITICAL]** 観点: セキュリティ
  - 場所: `<関数名・行番号>`
  - 問題: <問題の説明>
  - 改善案: <具体的な修正方法>

### 総評
<全体的な品質評価・特筆すべき点・マージ可否の判断>

重要度の定義:
- `CRITICAL`: マージ前に必ず対応が必要（セキュリティ脆弱性・バグ・重大な規約違反）
- `WARNING`: 対応を強く推奨（コード品質の低下・潜在バグ）
- `INFO`: 参考情報（リファクタリング提案・テスト追加推奨）

指摘がない場合は「指摘なし」と明示する。

レビューエージェントがレビュー全工程（差分取得・規約読込・レビュー・構造化出力）を自己完結で実行する。
結果が返却されるまで待機する。

### ステップ2: ユーザーへの確認

レビュー結果を提示したあと、以下を確認する:

- CRITICAL または WARNING の修正を実施するか
- 修正依頼があれば、以下の手順に準じて対応する:
  1. 対象ファイルを編集して修正を実施する
  2. 修正内容を英語・Conventional Commits形式でコミットする:
     ```bash
     git add <変更ファイル>
     git commit -m "$(cat <<'EOF'
     fix: address review issues

     - <対応内容1の英語要約>
     - <対応内容2の英語要約>

     Co-Authored-By: Claude <noreply@anthropic.com>
     EOF
     )"
     ```
  3. 修正内容と対応した指摘の一覧を日本語で報告する
